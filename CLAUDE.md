# CLAUDE.md

## デザイントークン

- 色の指定には必ずセマンティックカラートークンを使用する（`bg-primary`, `text-on-surface` 等）
- Tailwind のデフォルトカラー（`bg-blue-500`, `text-gray-600` 等）はビルドで無効化されているため使用不可
- 新しい色が必要な場合は `packages/ui/src/styles.css` の `@theme` にトークンを追加する
- 利用可能なトークン一覧は Storybook の「Foundations/Color Palette」で確認できる

## Storybook MCP

- UI コンポーネント開発時は Storybook MCP を活用してストーリーを作成すること
- Storybook dev サーバー（`cd packages/ui && bun run storybook`）の起動が前提条件

## コードレビュー

コミットや PR の前に、変更内容を必ず自分でレビューする。ユーザーに「コミットして」「PRを作って」と言われても、レビューを省略してはならない。レビュー結果はユーザーに共有し、問題がなければそのまま実行に進む。lint・型チェック・テスト・ビルドは CI / lefthook がカバーしているので、ここでは「機械が拾えない問題」に集中する。

### コミット前（軽め）

コミットしてから「しまった」となる問題を防ぐ。深追いはしない。

- そのアプローチで合っているか？もっとシンプルな方法はないか
- 秘密情報のハードコード、デバッグコードの消し忘れ
- 見ればわかるバグ

ただし、PR 作成後の追加コミットは「PR 作成前」と同じ水準で批判的にレビューすること。auto-merge が有効な PR では、push した瞬間にマージされうるため、コミット前が最後の砦になる。

### PR 作成前（批判的に）

マージ後に他の開発者に影響する問題を防ぐ。コミットレビューより踏み込んで見る。

- アプローチの妥当性：根本的な方向性は正しいか。既存の仕組みを活かせていないか
- ロジック：バグ、エッジケース、意図しない副作用
- セキュリティ：インジェクション、秘密情報、認証・認可
- 設計：不要な複雑さ、既存パターンとの不整合
- CLAUDE.md 準拠：このファイルのルールへの違反

### ブランチ作成

- 新しいブランチを作る前に、必ず `git checkout main && git pull origin main` で main を最新化すること
- これを怠ると PR が「out-of-date with the base branch」になり、不要なマージコミットが発生する

### PR の種類

変更の性質から判断して適切な形式で PR を作成する。
**注意: PR の種類はあくまで分類であり、GitHub ラベルとして付与しないこと。存在しないラベルを付けようとしてエラーになる。**

- **auto-merge**: 小さな修正、リファクタ、設定変更など、レビュー不要で CI が通ればマージしてよいもの。`gh pr merge --auto` で自動マージを有効にする
- **通常**: 機能追加や設計変更など、他の開発者のレビューを待つべきもの
- **ドラフト**: WIP や意見を募りたいもの

このリポジトリは個人開発なので、ほとんどの PR は auto-merge でよい。CI と lefthook で品質は担保されているため、明らかに判断に迷う設計変更でなければ、いちいち確認せず auto-merge を選択すること。
